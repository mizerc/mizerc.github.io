Next.js

Routing#

App Router#

In Next.js 13+, app route (/app) is part of the App Router, which uses the new file-based routing system with React Server Components. It supports layouts, templates, nested routing, loading/error states, and server/client components separation. This approach is more flexible and modern, designed for large-scale and dynamic apps.

The App Router is a file-system based router that uses React’s latest features such as Server Components, Suspense, and Server Functions.

Files#

What matters is having a page.ts for each page that you want to let Next.js to build for you. For API, you want the route.ts file. Next.js will create a route for each of those files that it can find and build the proper endpoint from the folder structure.

The layout.ts file can be used to hold header or any other logic that should stay as the user navigates in different pages.

Folder Structure#

app/
    page.tsx => /

    users/
        page.tsx => /users
    
    products/
        page.tsx => /products

    page4/
        page.tsx => /page4

    modules/
        auth/
            page.tsx => /modules/auth

    api/
        users/
            route.tsx => GET /api/users, POST /api/users
            [id]
                route.tsx => GET /api/users/3

Navigation#

<Link> is a React component provided by Next.js that wraps an <a> tag. It handles client-side navigation, prefetching, and is optimized for performance.

router.push() is an imperative function. You use it when you want to trigger navigation from logic, like inside a useEffect, a button click, or conditionally after some check. Lacks prefetch, SEO, and accessibility benefits.

Pages#

In contrast, pages route (/pages) is the traditional Pages Router, which uses file-based routing based on the pages/ directory. It supports basic routing, dynamic routes, and API routes but lacks the granular control and advanced features of the App Router.

Server-Side Rendering#

Server-side rendering (SSR) is a technique that renders a web page on the server rather than in the browser.

The app is fully rendered on the server and sent to the client. Then, the client’s JavaScript bundle engages and enables the Single Page Application framework to operate.

The fully rendered page contains all of the necessary information for them to see your site without having to wait for any JavaScript or CSS files to load.

Pros#

Fast load time

  • Small bundler sent to client
  • No need to wait the Browser to parse JavaScript and CSS to render the content.

Improved SEO

  • JavaScript can block Search and Social engines crawlers.

Better accessibility

  • Old devices can struggle with all the JavaScript required to run to render the page.

Cons#

  • You can’t deploy to a static hosting or CDN. You need a server/serverless backend to run the system that renders the page for each user that access your site.

  • Larger HTML size

Modularization#

  • Use barrel files (index.ts) to export module-specific functions

Development#

Boilerplate#

npx create-next-app@latest

Dev#

It uses turbopack to server in dev mode with support for watch changes and hot reloading.

npx next dev

npm run dev

Building#

After running npm run build, the output is placed in the .next directory at the root of your project. This folder contains all the compiled server and client code, including pre-rendered pages, static assets, and server functions. Then you can run npm run start to serve the prod build in your local computer.

Deploying#

You can deploy direct to Vite hosting. You can also host in a VPS. Or even static hosting if not using server components.

Error Boundary#

If you throw an error from a Client Component without catching it, Next.js will automatically trigger the closest error.tsx boundary in your route segment.

app/
    page.tsx

    error.tsx

    users/
        page.tsx
        error.tsx <== closest error boundary
